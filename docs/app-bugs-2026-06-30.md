## 1. Live daily pages predate the `ainotes` redesign — a sticker has no home  ·  **root cause**  ·  *tracked app-side as C5/D21*

The user's live daily pages under `Shared/2026-06/*` were built from an **old `daily` template**
that predates the 2026-06 region redesign:

- **No `data-fill`** on any region (the server derives fills by name as a fallback).
- **No `ainotes` and no `header` region.** The only region that derives to `fill: ai` is
  `affirmation`, and its box is **912×120**.

A habits-tracker sticker needs roughly **245×325** to be legible and checkable. There is **no
AI-owned region on these pages large enough to hold it** — so the sticker genuinely has no
legitimate home, and CoWork ended up placing it over the `schedule` (a `shared` region the user
writes in) or pushing it with a negative `y` up into the date/chrome band.

The **redesigned catalogue template already fixes this.** `Templates/daily-minimal/template.svg`
has a dedicated `ainotes` region — **289×422, `data-fill="ai"`**, `data-intent` = *"weather, the
day's context, an affirmation, plus a habit sticker or small image"* (weekend dailies are larger
still, ~288×708). The live pages simply weren't built from it.

**Live daily region geometry (for reference — `Shared/2026-06/2026-06-30`, page 1024×1366):**

| region | box w×h | derived fill |
|---|---|---|
| schedule | 540×870 | shared |
| priorities | 332×200 | shared |
| todo | 332×200 | shared |
| notes | 332×280 | ink |
| affirmation | 912×120 | **ai** (only AI-owned region — too short for a sticker) |

**Ask:** update the live daily template to the redesigned one (with `ainotes`), **or** provide an
in-app migration. The MCP can migrate only *empty* pages (see §4); the ink-bearing ones
(2026-06-25/26/27/28/30 carry 124–417 strokes each) need the app, because the new grid sits at
different coordinates than the old one and would misalign existing handwriting — the app is the
only place that can re-flow ink to the new geometry (or accept the shift knowingly).

## 2. Register `.folder.json → theme.accent` (or confirm `chromeAccent` may double)  ·  ✅ *resolved (app C4, 2026-07-02)*

To let a chapter carry an explicit body colour (e.g. lavender to-dos), the server now writes an
optional **`accent`** hex under `.folder.json → theme` and honours it in the underlay (tints body
text / markers / banners, floored dark for cream legibility). This is **additive** to the
`FORMAT.md §4` theme keys (`chromeAccent` / `harmony` / `varietyDial` / `fontPersonality`); the
server reads it defensively (absent ⇒ the gold/preset default), so nothing breaks before the app
catches up.

**Ask:** either (a) register `accent` as an optional `theme` key in `FORMAT.md §4` so app +
underlay agree on it, or (b) confirm the app is happy for the existing `chromeAccent` to double as
the shared accent (chrome *and* underlay) — in which case the server can map onto that instead.
Also confirm the app's `.folder.json → theme` parser **ignores unknown keys** (it should, but the
server writing `accent` relies on it).

## 3. On-device rendering — two verifications (currently unconfirmed)

Fold these into the `onionskin-mcp-open-verifications` checks:

1. **Does the renderer clip an `<image>` that overflows its region `<g>`, or draw it at its
   intended (possibly out-of-box) position?** ✅ **Answered (app C8, 2026-07-02): no clipping** —
   the renderer draws the image wherever it lands (`SVGCanvasRenderer.swift:68-73`). So a
   negative-`y` placement IS visible, and the server's `image_off_page` /
   `image_overlaps_region` warnings are the only guardrail against it.
2. **Does wrapped text render as expected on device?** ⬜ Still owed by the app (C8, second
   half). The server emits wrapped continuations as multiple stacked `<text>` elements at
   successive baselines (plain `<text>`, fully within the renderer's element set) — confirm the
   app draws all baselines as intended.
